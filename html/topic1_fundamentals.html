1. Large Language Models (LLMs)

What is an LLM?

A Large Language Model is a neural network trained on massive text corpora to predict the next token in a sequence. Despite this simple training objective, models at scale learn to reason, code, summarize, translate, and answer questions.

Key insight: LLMs are not databases. They don't "remember" facts — they learn statistical patterns. This distinction drives nearly every architectural decision in Agentic AI.

The Transformer Architecture (Simplified)

ComponentWhat it doesWhy it matters for agents
TokenizerSplits text into sub-word tokens (~100k vocab)Determines what fits in the context window
EmbeddingsMaps tokens to high-dimensional vectorsFoundation of semantic search in RAG
Self-AttentionEach token "attends" to all other tokensEnables long-range reasoning and instruction following
Feed-Forward LayersStore learned factual associationsSource of both knowledge and hallucinations
Context WindowMax tokens the model can process at onceHard limit on how much information an agent can use per call

Context Window — The Critical Constraint

The context window is the model's "working memory" — everything the model can see at once. Modern models range from 8K (older GPT-3.5) to 1M tokens (Gemini 1.5). But bigger isn't always better: cost and latency grow with context length, and models can "lose" information buried in long contexts.

Rule of thumb: Put the most important information at the start and end of the context. The "lost in the middle" problem is well-documented — models attend less to content in the middle of long prompts.

Prompting Strategies

StrategyWhen to useExample
Zero-shotSimple, well-defined tasks"Summarize this document in 3 bullet points."
Few-shotTasks needing format guidanceProvide 2-3 input→output examples before the task
Chain-of-Thought (CoT)Multi-step reasoning"Think step by step. First identify... then calculate..."
System promptPersona + behavior shaping"You are a senior software engineer. Be concise and technical."
ReActAgent reasoning with tool callsThought → Action → Observation loop (covered in Section 3)

from openai import OpenAI

print(response.choices[0].message.content)

2. Retrieval-Augmented Generation (RAG)

The Problem: LLMs Hallucinate

LLMs generate plausible-sounding text — even when they don't know the answer. This is called hallucination. It happens because:

  • The model's training data has a knowledge cutoff (no recent events)
  • Specialized or private data was never in the training set
  • The model fills gaps with statistically likely (but wrong) completions

The RAG Solution

RAG = Retrieval + Augmented Generation. Instead of asking the LLM to recall facts from memory, we:

  1. Index your documents as vector embeddings
  2. Retrieve the most relevant chunks at query time
  3. Augment the LLM prompt with retrieved context
  4. Generate a grounded, factual response
"RAG turns an LLM from a closed-book exam taker into an open-book exam taker. It still reasons — but now with the right information in front of it."

Embeddings and Vector Search

An embedding is a dense numerical vector (e.g., 1536 dimensions) that captures the semantic meaning of text. Two sentences with similar meaning have vectors that are close together in this high-dimensional space.

ConceptAnalogyTechnical detail
Embedding modelTranslator to "meaning space"e.g., text-embedding-3-small, sentence-transformers
Vector storeSemantic filing cabinetFAISS, Chroma, Pinecone, Weaviate
Similarity searchFind nearby neighborsCosine similarity or dot product over indexed vectors
Top-k retrievalGet the k closest documentsTypically k=3 to 5 chunks returned

RAG Pipeline Architecture

INDEXING PHASE (run once):
Documents → Chunking → Embedding Model → Vector Store

QUERY PHASE (run per question):
User Query → Embedding → Vector Store Search → Top-K Chunks
         → Prompt = "Context: {chunks}\n\nQuestion: {query}" → LLM → Answer

Building a RAG Pipeline in Python

from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain.chains import RetrievalQA

# Step 1: Load and chunk documents
loader = TextLoader("knowledge_base.txt")
docs = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(docs)

# Step 2: Embed and index
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# Step 3: Build RAG chain
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True
)

# Step 4: Query
result = qa_chain.invoke({"query": "What are best practices for LLM production?"})
print(result["result"])
print("\nSources:", [d.metadata for d in result["source_documents"]])

3. From LLM to Agent

What Makes Something an "Agent"?

An agent is an LLM that can observe, reason, and act in a loop until the task is complete. The key difference from a simple LLM call:

LLM CallAgent
Single prompt → single responseLoop: observe → reason → act → observe again
No toolsCan call tools (search, calculator, database, APIs)
StatelessMaintains memory across steps
Fixed outputSelf-corrects and retries on failure

The ReAct Pattern

ReAct (Reason + Act) is the foundational loop that powers most agents:

Thought: I need to find the current Python version.
Action: search_web("latest Python version 2025")
Observation: Python 3.13 was released in October 2024.

Thought: I now have the answer.
Action: FINAL ANSWER
Python 3.13 is the latest stable release.

The LLM generates Thought and Action. The framework intercepts the action, calls the actual tool, and feeds back the Observation. The loop continues until the LLM produces a final answer.

Core Agent Components

  • LLM Brain: The reasoning engine (GPT-4o, Claude, Gemini)
  • Tools: Python functions the LLM can call (search, calculator, database, code runner)
  • Memory: Short-term (conversation history) + Long-term (vector store)
  • Executor: The loop that runs Thought → Action → Observation

4. Connecting RAG with Agents

RAG as an Agent Tool

Instead of RAG being a fixed chain, we make it a tool the agent can choose to call. The agent decides when it needs retrieved context versus when it can answer from its own knowledge.

from langchain.tools import Tool
from langchain.agents import create_react_agent, AgentExecutor
from langchain import hub

# 1. Define RAG as a tool
rag_tool = Tool(
    name="knowledge_base_search",
    description="Search the company knowledge base for internal documents, policies, and procedures.",
    func=lambda query: qa_chain.invoke({"query": query})["result"]
)

# 2. Add other tools
from langchain_community.tools.tavily_search import TavilySearchResults
web_search = TavilySearchResults(max_results=3)

tools = [rag_tool, web_search]

# 3. Create ReAct agent
prompt = hub.pull("hwchase17/react")
llm = ChatOpenAI(model="gpt-4o", temperature=0)

agent = create_react_agent(llm=llm, tools=tools, prompt=prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 4. Run
result = executor.invoke({
    "input": "What is our company's refund policy, and how does it compare to industry standards?"
})
# Agent will: search KB for refund policy, then search web for industry standards
print(result["output"])

5. Full Working Example: Research Agent with RAG

import os
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.tools import Tool
from langchain.agents import create_react_agent, AgentExecutor
from langchain import hub

os.environ["OPENAI_API_KEY"] = "your-key-here"

# --- Build RAG ---
loader = TextLoader("knowledge_base.txt")
chunks = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)\
             .split_documents(loader.load())
vectorstore = FAISS.from_documents(chunks, OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

def rag_search(query: str) -> str:
    docs = retriever.invoke(query)
    return "\n\n".join(d.page_content for d in docs)

# --- Define tools ---
tools = [
    Tool(
        name="internal_search",
        func=rag_search,
        description="Search internal knowledge base for company-specific information."
    )
]

# --- Build agent ---
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
prompt = hub.pull("hwchase17/react")
agent = create_react_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True, max_iterations=5)

# --- Run ---
question = "What are the key benefits of our product according to internal docs?"
result = executor.invoke({"input": question})
print(result["output"])

6. Key Takeaways

LLM: A token-prediction engine that reasons through patterns learned from text. Not a database — grounding it with RAG is essential for factual accuracy.

RAG: Embed documents → store vectors → retrieve relevant chunks at query time → augment the prompt. Solves hallucination and knowledge cutoff problems.

Agent: LLM + Tools + Loop. The model reasons in a Thought→Action→Observation cycle, using tools (including RAG) to gather information and act.

RAG + Agent: The agent decides when to consult your knowledge base, when to search the web, and when it already knows the answer. This is the foundation of all production Agentic AI.
