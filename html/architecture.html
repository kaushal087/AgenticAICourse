Component Breakdown

1. User Interface Layer

Responsibilities:
- Accept natural language query from user
- Display intermediate reasoning steps
- Show final output

Implementation choices:
- Streamlit for interactive web UI (demo)
- CLI for headless/pipeline use
- FastAPI for production API integration

2. LangGraph Orchestrator

This is the brain of the system. It manages:
- The execution order of agents
- The shared state passed between agents
- Conditional routing (which agent runs next)
- Loop termination conditions

Core object:

from langgraph.graph import StateGraph, END

# Define the graph
workflow = StateGraph(GraphState)

# Add nodes (agents)
workflow.add_node("planner", planner_node)
workflow.add_node("researcher", researcher_node)
workflow.add_node("analyst", analyst_node)
workflow.add_node("writer", writer_node)

# Add edges (transitions)
workflow.set_entry_point("planner")
workflow.add_edge("planner", "researcher")
workflow.add_edge("researcher", "analyst")
workflow.add_conditional_edges(
    "analyst",
    should_continue,           # routing function
    {"continue": "researcher", # loop back for more research
     "done": "writer"}         # or proceed to writing
)
workflow.add_edge("writer", END)

app = workflow.compile()

3. Agent Nodes

Each agent is a Python function that takes GraphState and returns updated state fields.

Planner Agent

Input:  query (string)
Output: plan (list of 3-5 subtasks)
Role:   Decompose the user's query into actionable research questions
Model:  GPT-4o with structured output

Researcher Agent

Input:  plan (list of subtasks)
Output: research_results (list of {subtask, findings})
Role:   For each subtask, retrieve relevant information
Tools:  Web search (Tavily), RAG retrieval (FAISS)
Model:  GPT-4o with tool calling

Analyst Agent

Input:  research_results
Output: analysis (synthesized insights), approved (bool)
Role:   Evaluate quality, identify gaps, synthesize findings
Logic:  If coverage < threshold → loop back to Researcher
Model:  GPT-4o with critic prompt

Writer Agent

Input:  analysis, research_results, query
Output: final_report (markdown string)
Role:   Produce a well-structured, cited final report
Model:  GPT-4o with structured writing prompt

4. Tool Layer

Tool Library Purpose

5. RAG Pipeline

Retrieval-Augmented Generation enhances agent responses with private/domain knowledge.

┌─────────────────────────────────────────────┐
│                  RAG PIPELINE               │
│                                             │
│  Documents → Chunker → Embedder → FAISS     │
│                                     │       │
│  Query → Embedder → Similarity Search       │
│                          │                  │
│                    Top-K Chunks             │
│                          │                  │
│                    LLM + Context → Answer   │
└─────────────────────────────────────────────┘

Step by step:
1. Ingest: Load documents (PDF, text, web)
2. Chunk: Split into ~500-token pieces with overlap
3. Embed: Convert to vectors using text-embedding-3-small
4. Index: Store in FAISS vector store
5. Retrieve: On query, find top-K most similar chunks
6. Augment: Add chunks to LLM context
7. Generate: LLM answers using retrieved context

Scaling Considerations

Concern Solution
Latency Run researcher subtasks in parallel (LangGraph parallel nodes)
Cost Cache embeddings; use GPT-4o-mini for non-critical agents
Reliability Add retry logic per node; checkpoint graph state
Security Sanitize tool inputs; rate-limit API calls
Observability LangSmith tracing for all node executions

Design Decisions & Tradeoffs

Why LangGraph over CrewAI?

Feature LangGraph CrewAI
Control Full Python control Higher-level abstraction
Flexibility Arbitrary graph topology Role-based delegation
Debugging Explicit state inspection More opaque
Learning curve Medium Lower
Production readiness ✅ Checkpointing, streaming ⚠️ Less mature

Choose LangGraph when: you need fine-grained control over workflow logic.
Choose CrewAI when: you want rapid prototyping with role-based agents.

Why FAISS over Chroma/Pinecone?

  • FAISS = local, in-memory, no external service needed → perfect for demos
  • Chroma = local persistent store → good for development
  • Pinecone = managed cloud vector DB → production at scale
