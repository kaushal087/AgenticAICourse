🧠 Core Concepts (3-Minute Version)

The Mental Model

Think of your engineering team:

You (Tech Lead):     "Here's the task breakdown" → PLANNER AGENT
Senior Dev:          "I'll research this"         → RESEARCHER AGENT  
Code Reviewer:       "Let me check the quality"   → ANALYST AGENT
Technical Writer:    "I'll document the output"   → WRITER AGENT

Key Terms

Term Meaning Analogy
Agent LLM + tools + loop A developer with Google + IDE
LangGraph Node Python function = one agent One team member's job
GraphState Shared data between agents The shared Google Doc
Edge Transition between agents Handoff between team members
Conditional Edge Branch based on state Manager decision: "needs more work? or ship it?"

🔧 System Components Deep Dive

1. GraphState (Shared Memory)

class GraphState(TypedDict):
    query: str              # Original question
    plan: List[str]         # Subtasks from Planner
    research_results: List  # Findings from Researcher  
    analysis: str           # Summary from Analyst
    final_report: str       # Output from Writer
    iteration_count: int    # Loop guard (max 3)
    approved: bool          # Analyst's verdict

2. The Graph Definition

workflow = StateGraph(GraphState)
workflow.add_node("planner",    planner_node)
workflow.add_node("researcher", researcher_node)
workflow.add_node("analyst",    analyst_node)
workflow.add_node("writer",     writer_node)

workflow.set_entry_point("planner")
workflow.add_edge("planner", "researcher")
workflow.add_edge("researcher", "analyst")
workflow.add_conditional_edges("analyst", should_continue,
    {"continue": "researcher", "done": "writer"})
workflow.add_edge("writer", END)

What happens when user asks: "Analyze AI's impact on SWE careers"

⏱ t=0s  │ User submits query
         │
⏱ t=1s  │ PLANNER: query → plan
         │   "What SWE tasks are being automated?"
         │   "What do industry surveys say?"
         │   "What new roles are emerging?"
         │
⏱ t=8s  │ RESEARCHER: plan → research_results
         │   For each subtask:
         │     search_web(subtask) → snippets

         │     summarize combined findings
         │

         │
⏱ t=20s │ WRITER: analysis + results → final_report
         │   "# Impact of AI on Software Engineering..."
         │
⏱ t=21s │ User receives full report with sources

💻 Code Walkthrough — Live Demo

File: code/agent.py — Planner Node

def planner_node(state: GraphState) -> dict:
    """Break user query into focused research subtasks."""

    prompt = f"""You are a research planner.
    Break this query into 3-5 specific research questions:
    Query: {state['query']}

    Return JSON: {{"plan": ["question1", "question2", ...]}}"""

    response = llm.invoke([SystemMessage(content=prompt)])
    data = json.loads(response.content)

    print(f"📋 Plan created: {len(data['plan'])} subtasks")
    return {"plan": data["plan"]}

Key Design Decisions

  1. Structured output — JSON response ensures parseable plan
  2. Single responsibility — Planner only plans, never researches
  3. State update — returns ONLY what changed (plan)
  4. Print statements — transparency for debugging & demo

🌍 Real-World Applications

Use Case 1: Automated Due Diligence (Finance)

Planner   → Break "Analyze company X" into financial, legal, market questions
Researcher → Pull SEC filings, news, LinkedIn data
Analyst   → Flag risks, verify coverage
Writer    → Generate DD report with confidence scores

Use Case 2: Autonomous Code Review (DevTools)

Planner   → Identify review aspects: security, performance, style
Researcher → Fetch similar code patterns, CVE database, style guides
Analyst   → Score against criteria
Writer    → Generate PR review comment with suggestions

Use Case 3: Clinical Literature Review (Healthcare)

Planner   → Break clinical question into PubMed queries
Researcher → Retrieve papers, extract findings
Analyst   → Grade evidence quality (GRADE framework)
Writer    → Generate clinical summary with citations

The Pattern is Universal

Planner → Researcher → Validator → Synthesizer
Works in any domain requiring complex information gathering + synthesis.

⚖️ Tradeoffs & Limitations

When Multi-Agents WIN

✅ Use Multi-Agents When ❌ Don't Use When
Task has clear sub-problems Simple Q&A
Quality validation needed Single-step retrieval
Parallel research possible Real-time chat (< 2s needed)
Specialized expertise required Batch classification

Known Challenges

1. Latency
Multi-agent runs take 15–30s. Not suitable for conversational UX without streaming.

2. Cost
Multiple LLM calls per query. Estimate: 4 agents × 3 tool calls × GPT-4o ≈ $0.05–$0.20/query.

3. Error Propagation
If Planner produces a bad plan, all downstream agents are affected. Always validate Planner output.

4. LLM Non-Determinism
Same query can yield different plans. Use temperature=0 for Planner/Analyst; higher for Writer.

5. Debugging Complexity
Failures can occur at any node. Use LangSmith traces or verbose state printing for diagnosis.

The 3 Big Takeaways

  1. "Use agents when task decomposition provides real value"
    Not every problem needs multi-agents. Identify when it helps.

  2. "Shared state makes systems inspectable"
    The GraphState is the debuggable, resumable backbone.

  3. "The Critic/Analyst pattern is what makes agents reliable"
    Without a quality gate, agents hallucinate confidently.
