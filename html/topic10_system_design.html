🏗️ System Design for AI Applications

This topic covers architectural patterns and design principles for building scalable, reliable, and production-grade AI systems. Understanding these patterns is essential for senior engineering roles and AI system design interviews.

Why System Design Matters for AI Engineers

AI systems face unique design challenges compared to traditional software:

  • Non-determinism: LLM outputs are probabilistic — you must design for variability
  • Latency: LLM inference is slow (1–30 seconds) — caching and async patterns are critical
  • Cost: Every token costs money — efficient prompt design and caching reduce expenses significantly
  • Observability: Traditional logging is insufficient — you need trace-level visibility into agent reasoning
  • Safety: Output moderation, hallucination detection, and guardrails are first-class concerns

Core Architectural Patterns

1. Single-Agent Architecture

The simplest pattern: one LLM with tools, a system prompt, and memory.

User Input
    │
    ▼
┌─────────────────────────┐
│  Agent (LLM + Tools)    │
│  ┌──────────────────┐   │
│  │  System Prompt   │   │
│  │  Tool Registry   │   │
│  │  Memory Store    │   │
│  └──────────────────┘   │
└────────────┬────────────┘
             │
             ▼
         Response

Use when: Single domain, low complexity, <5 tools, response time < 10s acceptable.

2. Pipeline Architecture (Chain)

Sequential processing through specialized stages.

Input → [Preprocessor] → [LLM Chain] → [Postprocessor] → Output
                              │
                         [Tool Calls]

Use when: Fixed workflow with no branching, ETL-style data transformation.

3. Multi-Agent Supervisor Architecture

A supervisor agent orchestrates specialist agents.

User Query
    │
    ▼
┌──────────────┐
│  Supervisor  │ ←── decides routing
└──────┬───────┘
       │
   ┌───┴──────────────────┐
   │           │           │
   ▼           ▼           ▼
[Research]  [Coding]  [Analysis]
  Agent      Agent     Agent
   │           │           │
   └───────────┴───────────┘
               │
               ▼
          Final Response

Use when: Multiple domains, dynamic routing, parallel execution needed.

4. Hierarchical Multi-Agent Architecture

Multiple tiers of agents — supervisors managing sub-supervisors.

Orchestrator
├── Supervisor A
│   ├── Agent A1 (specialized)
│   └── Agent A2 (specialized)
└── Supervisor B
    ├── Agent B1 (specialized)
    └── Agent B2 (specialized)

Use when: Enterprise-scale tasks, thousands of subtasks, team-level organization needed.

5. Event-Driven Agent Architecture

Agents triggered by events from queues or streams.

Event Sources (Kafka, SQS, webhooks)
    │
    ▼
Event Router
    │
    ├──▶ Agent Pool (scales horizontally)
    │         │
    │    [Process Event]
    │         │
    └──▶ Result Store → Downstream Systems

Use when: High throughput, asynchronous processing, real-time data pipelines.

Scalability & Performance

Caching Strategies

  • Semantic caching: Cache by query embedding similarity (e.g., GPTCache). Reduces LLM calls by 30–70% for repetitive queries.
  • Prompt caching: OpenAI and Anthropic cache repeated system prompts. Reduces cost for long system prompts.
  • Result caching: Redis-based exact-match caching for deterministic queries.

[Redis State Store]  [Vector Store]

Keep agent workers stateless — externalize state to Redis or a database so any worker can handle any request.

Reliability & Fault Tolerance

Retry and Fallback Patterns

# Exponential backoff with fallback model
async def call_llm_with_retry(prompt, max_retries=3):
    for attempt in range(max_retries):
        try:
            return await primary_model.ainvoke(prompt)
        except RateLimitError:
            await asyncio.sleep(2 ** attempt)
        except ModelUnavailableError:
            return await fallback_model.ainvoke(prompt)  # downgrade
    raise MaxRetriesExceeded()

Circuit Breaker for Agent Tools

Prevent cascading failures when external APIs (search, databases) are slow or down:

  • Closed: Normal operation, calls pass through
  • Open: Failures exceeded threshold, calls fail fast with cached/default response
  • Half-open: Probe to check if service recovered

Security Considerations

Prompt Injection Defense

Prompt injection is the #1 threat for LLM applications — malicious content in retrieved documents or user input can hijack agent behavior.

  • Separate system instructions from user/external content using delimiters and roles
  • Validate tool call arguments before execution — never execute arbitrary code from LLM output
  • Use allowlists for tool names and parameters
  • Apply output moderation before returning to users

Data Privacy

  • PII detection and redaction before sending to LLM APIs
  • Tenant isolation in multi-tenant RAG systems (per-tenant namespaces in vector stores)
  • Audit logging of all LLM inputs and outputs for compliance

System Design Interview Framework

When asked to design an AI system in an interview, use this structured approach:

  1. Clarify requirements — scale, latency SLA, accuracy requirements, budget, data privacy
  2. High-level design — choose single-agent vs multi-agent, RAG vs fine-tuning, sync vs async
  3. Data flow — ingestion pipeline, retrieval, LLM call, post-processing
  4. Scale bottlenecks — where will it break at 10×, 100× load?
  5. Reliability — retry logic, fallbacks, graceful degradation
  6. Cost optimization — caching, model tiering, batching
  7. Monitoring — what metrics prove the system is healthy?

Key Takeaways

  • Choose the simplest architecture that meets your requirements — don't over-engineer
  • Design for observability from day one — you cannot improve what you cannot measure
  • Keep agent workers stateless — externalize all state
  • Implement semantic caching to reduce both latency and cost in production
  • Treat prompt injection as a security vulnerability, not an edge case
  • Always design a graceful degradation path when LLMs fail or respond poorly
